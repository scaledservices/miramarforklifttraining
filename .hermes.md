# Miramar Repo — Hermes Project Context (`.hermes.md` draft)

This file governs Hermes when working in the `scaledservices/miramarforklifttraining` repo. It is a snapshot from June 30, 2026 — verify current repo/site state before trusting any specific claim below.

## Project Identity

This repo supports the Miramar training-site rebuild/improvement effort. **Phase 1 objective: Miramar-first revenue growth through `training.miramarforklift.com`** — more new-customer leads, quote requests, calls, booked trainings, attributable revenue.

**ForkliftCertified is parked.** It was this repo's original white-label brand vision. Do not optimize for it, do not revive it, do not treat ForkliftCertified-branded code as current direction unless the user explicitly reopens that project.

## Business Priority (strict order)

1. **Onsite/company training** (trainer travels to customer facility) — highest revenue per person, highest priority.
2. **Hands-on/location training** (customer travels to Miramar/rented facility) — second priority.
3. **Online certification/renewal** — fallback path only: renewals, experienced operators, out-of-area customers. Do not visually or structurally favor this over 1 and 2 (no "Most Popular" badges, no top-of-funnel default).

## Scope Boundaries

**In scope** (deploy still needs explicit user approval):
- Training-site funnel, quote request flow, product/location pages
- SEO/GEO/AEO content and structure
- Lead capture, analytics/attribution
- CRM/attribution ledger foundation (source tracking — not an authoritative customer DB)

**Out of scope without explicit, separate approval:**
- `miramarforklift.com` (the main company site, distinct from the training subdomain)
- Real payment processing changes, Authorize.net credentials/flow
- Certificate/license-card issuance changes
- Importing customer databases as an authoritative system of record
- Production deploys
- ForkliftCertified launch
- Claims beyond Miramar's existing, approved offerings

If a task approaches any of these, stop and ask before proceeding — even if the change looks small.

## Repo-Specific Warnings (verify current state before relying on these)

- **Payments**: existing Stripe/`stripe-replit-sync` path is hard-locked to Replit's Connectors API and is not Phase 1 priority. Do not assume Stripe is the right payment path for Miramar — Miramar actually uses Authorize.net, with check/ACH/card (3% fee) as the real-world payment flow.
- **Earnings logic incomplete**: `payments.platformEarnings`/`partnerEarnings` columns exist but are never populated by any payment code path. The profitability dashboard will show $0 for real transactions until this is actually wired up.
- **Location config likely conflicts with business priority**: `shared/config/locations.ts` marks Las Vegas and Fresno `active: false` as of last review, despite Las Vegas being Alberto's best-performing inbound market. Confirm current state before assuming this is fixed.
- **Homepage/positioning likely over-promotes online certification** (e.g. "Most Popular" badge), contradicting the priority order above. Confirm current state before assuming this is fixed.
- **Security cleanup required before public deployment**: `cookies.txt` with a live session cookie was committed to the repo; `server/generated-pdfs/` contains committed certificate/invoice PDFs that may be real or test customer data. Resolve before any public-facing deploy.

## Required Workflow Before Implementation

1. Read the vault index (`Companies/Miramar/INDEX.md`) and the current growth plan before planning any task.
2. Verify current repo state directly before editing — vault artifacts (repo review, site audits) are dated snapshots, not live monitoring.
3. Use a focused branch per task; never push to `main` without explicit user approval.
4. Run available quality gates (typecheck/build/tests/lint — inspect `package.json` scripts first) before declaring work complete.
5. For anything touching payments, certificates, customer data, or production, stop and ask for explicit approval before proceeding.

## Quality Expectations

- Favor small, reviewable changes over large speculative ones.
- Prioritize revenue-impacting work (funnel, lead capture, positioning fixes) over platform/architecture work that doesn't move the Phase 1 goal.
- Do not overbuild — this is a lean revenue-growth effort, not a platform rebuild for its own sake.
- Preserve existing bilingual (EN/ES) and SEO infrastructure where it's already functional; don't discard working structure to rebuild from scratch.
- Never introduce fake testimonials, fake reviews, fabricated statistics, or unsupported compliance/certification claims. Real content only.
